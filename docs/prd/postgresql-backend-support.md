# PRD: PostgreSQL Backend Support

## Overview

Add PostgreSQL as an alternative metadata database backend for Data-Forge, allowing users to choose between SQLite (default, zero-config) and PostgreSQL (scalable, multi-instance) for both the shared settings database and per-namespace databases.

## Problem Statement

Data-Forge currently uses SQLite exclusively for metadata storage:

- **Settings database** (`app.db`): Stores users, sessions, app settings — shared across namespaces.
- **Namespace databases** (`namespace.db`): Per-namespace storage for analyses, datasources, engine runs, schedules, etc.

SQLite works well for single-instance deployments but becomes a limitation when:

- Multiple app instances need to share state (load balancing, HA).
- The namespace database grows large and WAL mode contention increases.
- Users want centralized database backups with standard tooling (pg_dump).
- S3 storage mode is enabled (SQLite files cannot live on S3).

### Current State

| Concern | Status |
|---------|--------|
| SQLite settings DB | ✅ `app.db` at `DATA_DIR` root |
| SQLite namespace DBs | ✅ `namespace.db` per namespace, LRU cached (max 50) |
| PostgreSQL for external datasources | ✅ Read-only via `pl.read_database()` |
| PostgreSQL for app metadata | ❌ Not supported |
| Alembic migrations | ✅ SQLite only (batch mode) |
| Session management | ✅ SQLite-specific (sync engine, WAL pragma) |

### Target State

| Concern | Status |
|---------|--------|
| SQLite settings + namespace DBs | ✅ Unchanged (default) |
| PostgreSQL settings + namespace DBs | ✅ Single PostgreSQL instance, schema-per-namespace |
| Alembic migrations | ✅ Dialect-aware (SQLite batch mode, PostgreSQL native DDL) |
| Connection pooling | ✅ SQLAlchemy pool for PostgreSQL, NullPool for SQLite |
| Async support | ✅ `asyncpg` driver for PostgreSQL |

## Goals

| # | Goal | Success Metric |
|---|------|----------------|
| G-1 | PostgreSQL as metadata backend | All CRUD operations work identically with PostgreSQL |
| G-2 | Schema-per-namespace isolation | Each namespace maps to a PostgreSQL schema, preserving multi-tenancy |
| G-3 | Zero-change for SQLite users | Existing SQLite deployments work without any configuration changes |
| G-4 | Dialect-aware migrations | Alembic generates correct DDL for both SQLite and PostgreSQL |
| G-5 | Connection pooling | PostgreSQL uses connection pool; SQLite uses NullPool (unchanged) |

## Non-Goals

- MySQL/MariaDB support (PostgreSQL only for now)
- Automatic migration from SQLite to PostgreSQL (provide a manual guide/script)
- Read replicas or connection routing
- Multi-database PostgreSQL (single database, multiple schemas)

## User Stories

### US-1: Configure PostgreSQL as Metadata Backend

> As a user deploying on a team server, I want to use PostgreSQL for all metadata so multiple instances can share state.

**Acceptance Criteria:**

1. Setting `DATABASE_URL=postgresql+psycopg://user:pass@host:5432/dataforge` switches all metadata storage to PostgreSQL.
2. On first startup with PostgreSQL, the schema is created via Alembic migrations.
3. Settings database tables live in the `public` schema.
4. Namespace-specific tables live in `ns_{namespace}` schemas (e.g., `ns_default`).
5. The app validates the connection on startup and fails fast with a clear error if unreachable.

### US-2: Run Migrations on PostgreSQL

> As a user upgrading Data-Forge, I want Alembic migrations to run correctly against PostgreSQL.

**Acceptance Criteria:**

1. Existing migrations are updated to be dialect-aware: SQLite uses batch mode, PostgreSQL uses native ALTER TABLE.
2. `migrate.sh` detects the database dialect and applies the correct strategy.
3. New migrations generated by developers work for both dialects.
4. Rollback (`alembic downgrade`) works for both dialects.

### US-3: Namespace Isolation with PostgreSQL Schemas

> As a user with multiple namespaces, I want each namespace's data isolated in PostgreSQL.

**Acceptance Criteria:**

1. Each namespace gets a PostgreSQL schema: `ns_{namespace_name}`.
2. Schema is created automatically on first access (CREATE SCHEMA IF NOT EXISTS).
3. SQLAlchemy sessions use `search_path` set to the namespace schema.
4. Cross-namespace queries are not possible through normal app operations.
5. Dropping a namespace removes its schema (CASCADE).

## Technical Design

### Database URL Detection

```python
# core/config.py
@property
def is_postgres(self) -> bool:
    return self.database_url.startswith("postgresql")

@property
def is_sqlite(self) -> bool:
    return self.database_url.startswith("sqlite")
```

### Engine Factory

```python
# core/database.py
def _create_engine(url: str, schema: str | None = None) -> Engine:
    if url.startswith("postgresql"):
        connect_args = {"options": f"-csearch_path={schema}"} if schema else {}
        return create_engine(
            url,
            pool_size=5,
            max_overflow=10,
            pool_pre_ping=True,
            connect_args=connect_args,
        )
    else:
        # Existing SQLite engine creation with WAL pragma
        ...
```

### Namespace Schema Management

For PostgreSQL, namespaces map to schemas rather than separate database files:

| SQLite | PostgreSQL |
|--------|-----------|
| `data_dir/app.db` | `public` schema |
| `data_dir/namespaces/{ns}/namespace.db` | `ns_{ns}` schema |

The `get_namespace_engine()` function:
- **SQLite**: Creates/caches an engine per `namespace.db` file (existing behavior).
- **PostgreSQL**: Creates/caches an engine per schema with `search_path` set.

### Alembic Dialect Awareness

Update `env.py` to branch on dialect:

```python
def run_migrations_online():
    context.configure(
        connection=connection,
        target_metadata=target_metadata,
        render_as_batch=connection.dialect.name == "sqlite",
    )
```

For PostgreSQL namespace migrations, run Alembic against each schema:

```python
for schema in get_all_namespace_schemas(connection):
    connection.execute(text(f"SET search_path TO {schema}"))
    context.run_migrations()
```

### Model Compatibility

All existing models use SQLAlchemy/SQLModel with standard types. Potential incompatibilities:

| SQLite | PostgreSQL | Action |
|--------|-----------|--------|
| `JSON` column | `JSONB` | Use `sa.JSON` — SQLAlchemy handles dialect mapping |
| `BOOLEAN` as INTEGER | Native BOOLEAN | No change — SQLAlchemy handles it |
| Auto-increment INTEGER PK | SERIAL/IDENTITY | No change — SQLAlchemy handles it |
| `datetime` as TEXT | Native TIMESTAMP | Use `sa.DateTime` — already in use |
| WAL pragma | N/A | Skip pragma for PostgreSQL |

### Dependencies

| Dependency | Purpose |
|-----------|---------|
| `psycopg[binary]` | PostgreSQL driver (sync) |
| `asyncpg` | PostgreSQL driver (async, if needed) |
| `psycopg` already in SQLAlchemy URL | `postgresql+psycopg://` dialect |

### Migration Script (SQLite → PostgreSQL)

Provide a utility script:

```bash
data-forge db migrate-to-postgres \
  --source sqlite:///data/app.db \
  --target postgresql+psycopg://user:pass@host/db
```

Steps:
1. Connect to both databases.
2. Create schemas for each namespace.
3. Copy tables row-by-row with type coercion.
4. Validate row counts match.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| SQL dialect differences in raw queries | Medium | High | Audit all raw SQL; use SQLAlchemy ORM/Core exclusively |
| Schema creation race condition | Low | Medium | Use `CREATE SCHEMA IF NOT EXISTS` with advisory lock |
| Connection pool exhaustion | Medium | Medium | Configure pool size relative to worker count; expose `DB_POOL_SIZE` env var |
| Migration compatibility across dialects | Medium | High | CI tests run against both SQLite and PostgreSQL |
